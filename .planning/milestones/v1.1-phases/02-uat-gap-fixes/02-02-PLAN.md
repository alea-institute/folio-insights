<objective>
Fix UAT Issue I-2: `POST /api/v1/corpus/{id}/export/bundle` returns HTTP 422 with an empty body when the corpus has no approved tasks, because FastAPI's Pydantic body validation fires before the handler can raise the 404. Sibling endpoints (`owl`, `ttl`, `jsonld`, `validation`) all return 404 with `{"detail":"No approved tasks to export"}`. Make bundle match.

Purpose: Contract parity across all export endpoints. Clients can rely on 404 == no data; 422 only means malformed body.
Output: `export_bundle` endpoint returns 404 (same detail string) when there are no approved tasks, whether or not a body was supplied. New pytest regression test locks the behavior in.
</objective>

The bug: FastAPI parses `body: BundleRequest` BEFORE entering the handler body, so a missing/empty body raises 422 before the 404 check runs. The 422 happens even if the corpus has zero approved tasks.

Sibling 404 detail strings (already in the codebase — MUST match verbatim):
- `export_owl`: `HTTPException(status_code=404, detail="No approved tasks to export")` (line ~168)
- `export_ttl`: same (line ~194)
- `export_jsonld`: same (line ~220)
- `export_validation`: same (line ~248)
- `_load_export_data` raises `detail="No discovered tasks found"` (line 52) when there are no task rows AT ALL

Note the distinction: `_load_export_data` returns 404 with "No discovered tasks found" when the `task_decisions` table is empty. `_get_approved_tasks` returns `[]` and the handler raises 404 "No approved tasks to export" when there are tasks but none are `status='approved'`. Both are legitimate 404s — the bundle endpoint should emit whichever is relevant, matching sibling behavior.
