## Human Verification Required

### 1. End-to-End Upload and Process Workflow

**Test:** Start the backend (`python -m api.main`, port 8700) and frontend dev server (`cd viewer && npm run dev`). Open http://localhost:5173/upload. Create a corpus, drag a .md file onto the upload zone, verify it appears in the file list, click "Process Corpus", and watch the UI.
**Expected:** Progress bar advances, stage pills turn blue (active) then green (complete) as each of 7 stages runs, the activity log populates with timestamped entries, and after ~1.5s of the complete state the browser navigates to `/?corpus=<id>`.
**Why human:** SSE EventSource state transitions, real-time DOM updates, and delayed navigation require a live browser with the actual pipeline executing.

### 2. ZIP Archive Upload and Extraction

**Test:** Upload a .zip file containing 2–3 supported documents (e.g., .md and .txt files).
**Expected:** The upload zone accepts the file, the backend extracts it server-side, and the individual extracted files appear in the file list (not the zip itself).
**Why human:** Server-side extraction behavior with a real HTTP multipart upload requires a live session.

### 3. Corpus Create and Delete Dialogs

**Test:** Click "+ New Corpus", type the name of an existing corpus, confirm. Then try again with a unique name.
**Expected:** First attempt shows validation error "A corpus with this name already exists." Second attempt creates the corpus and auto-selects it. Then delete it — the dialog shows the corpus name in bold, and after confirm the corpus is gone from the sidebar.
**Why human:** Dialog focus trap, real-time validation, and dynamic DOM state require browser interaction.

### 4. Auto-Navigation to Review Page on Completion

**Test:** Process a corpus to completion and observe the transition.
**Expected:** Green progress bar and all-green stage pills for ~1.5s with "Processing complete — N files processed, M knowledge units extracted." message, then automatic navigation to `/?corpus=<id>` with that corpus pre-selected in the header dropdown.
**Why human:** Timing, routing, and store state synchronization require a live browser session.

### 5. Folder Upload (webkitdirectory)

**Test:** Click "or upload a folder" inside the upload zone.
**Expected:** A folder-picker dialog opens; selecting a folder uploads all supported files from its root.
**Why human:** `webkitdirectory` is browser-native and has no programmatic equivalent in static analysis.
