# Phase 1.1: Upload & Processing UI - Context

**Gathered:** 2026-03-17 (updated)
**Status:** Ready for planning

<domain>
## Phase Boundary

Add document upload, corpus management, and pipeline processing to the existing web UI. Users upload files through the browser, manage named corpora, trigger extraction processing, watch real-time progress, and auto-navigate to the review viewer on completion. This replaces the current CLI-only workflow (`folio-insights extract <dir>`) with a browser-based equivalent.

</domain>

<decisions>
## Implementation Decisions

### Upload mechanism
- Drag-and-drop zone AND file picker (both available)
- Accept individual files in all 14 supported formats (MD, TXT, DOCX, PDF, HTML, RTF, EML, MSG, XML, CSV, XLSX, TSV, WPD)
- Accept bulk uploads: zip archives (extracted server-side) and folder selection (via `webkitdirectory`)
- Use standard multipart/FormData for file upload (more efficient for large/bulk files than folio-enrich's base64-in-JSON approach)

### Upload page placement
- Separate page/tab in the viewer — not integrated into the existing three-pane review layout
- Navigation between Upload and Review pages via header tabs/nav

### Upload page layout
- Two-column layout: corpus list sidebar (left) + upload zone & file list (right)
- File list shows minimal info: filename, format icon, processing status (pending/done/error)
- "Process" button positioned above the file list — prominent call-to-action between upload zone and file list
- During processing, progress display (stage pills + activity log) replaces the upload zone — focused view, upload disabled while processing

### Corpus management
- Users can create, name, and delete corpora
- Each corpus has its own uploaded files directory and extraction output
- Corpus selector in header (already exists in review viewer — extend it)
- Corpus list page showing name, file count, processing status, last processed date

### Processing trigger
- "Process" button (not auto-process on upload)
- Processes ALL unprocessed files in the corpus — the existing corpus registry tracks by SHA-256 hash, so previously processed files are naturally skipped
- User can upload multiple batches before pressing Process

### Progress display
- SSE (Server-Sent Events) matching folio-enrich's pattern: `GET /api/v1/corpus/{id}/stream`
- Stage-by-stage progress bar with stage pills (ingestion → structure → boundary → distillation → classification → folio tagging → deduplication → quality gating → output)
- Collapsible real-time activity log with timestamped messages (stage name + message)
- Modeled after folio-enrich's SSE implementation in `backend/app/services/streaming/sse.py`

### Post-processing flow
- On completion, auto-navigate to the review viewer with the processed corpus selected
- Transition should feel seamless — user sees progress finish, then lands in the review viewer ready to review

### Claude's Discretion
- Exact progress bar visual design (match existing dark theme)
- Upload zone styling and drag states
- Error handling UX for failed uploads or processing errors
- Activity log formatting and collapsibility
- File management within corpus (remove individual files, etc.)
- Corpus lifecycle details (re-processing, renaming)

</decisions>
