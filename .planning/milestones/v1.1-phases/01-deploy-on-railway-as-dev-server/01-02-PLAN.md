  <action>
    Create `railway.toml` at repo root with the exact contents below. This pins:
    - Builder: DOCKERFILE (uses the Dockerfile we created in Plan 01)
    - Healthcheck: /health endpoint (added in Plan 01 Task 1)
    - Restart: on failure, up to 3 times
    - Numerical limits kept at Railway defaults (no need to override for a dev server)

    ```toml
    # Railway configuration for folio-insights dev deploy.
    # See https://docs.railway.com/reference/config-as-code
    #
    # Scope: single-service dev deployment (Phase 1).
    # Prod hardening (custom domain, replicas, auth) is out of scope.

    [build]
    builder = "DOCKERFILE"
    dockerfilePath = "Dockerfile"

    [deploy]
    # Railway polls this path after the container boots. If it doesn't return
    # HTTP 2xx within healthcheckTimeout, the deploy is rolled back.
    healthcheckPath = "/health"
    healthcheckTimeout = 120

    # Restart on unexpected exit (OOM, uncaught exception). Keep retries small;
    # persistent crash should surface as a failed deploy, not a restart loop.
    restartPolicyType = "ON_FAILURE"
    restartPolicyMaxRetries = 3
    ```

    Do NOT add a `startCommand` — the Dockerfile's CMD already handles that and uses `${PORT:-8000}` correctly. Adding one here would bypass the Dockerfile's CMD and the $PORT substitution.

    Do NOT add a `numReplicas` — Railway default is 1 for hobby/dev.

    Do NOT add watch paths — default behavior (redeploy on any push to the linked branch) is fine.
  </action>
  <verify>
    <automated>test -f railway.toml && grep -q 'builder = "DOCKERFILE"' railway.toml && grep -q 'healthcheckPath = "/health"' railway.toml && grep -q 'restartPolicyType' railway.toml && echo OK</automated>
  </verify>
