## Milestone Summary

**Key Decisions:**
- [Phase 01-01] Bundled `output/` into image rather than Railway volume — simplest dev path per CONTEXT
- [Phase 01-01] Multi-stage Docker pattern mirrors folio-enrich + folio-mapper (node:20-slim builder → python:3.11-slim runtime + uv)
- [Phase 01-01] Belt-and-suspenders install of fastapi + uvicorn[standard] + python-multipart on top of `uv pip install .` — guards against pyproject transitive-dep drift
- [Phase 01-02] `railway.toml` omits `startCommand` so Dockerfile's `${PORT:-8000}` substitution is preserved; `healthcheckTimeout=120s` to accommodate heavy torch+sentence-transformers boot
- [Phase 01-03] Whitelisted `output/default/` + `output/test1/` in `.gitignore` so bundled corpora ship in the Railway build context
- [Phase 02-03] FOLIO label-match threshold lowered `0.7 → 0.6`, exposed as class-level `_FOLIO_LABEL_RESOLUTION_THRESHOLD`; unresolved labels rewritten to `extraction_path="proposed_class"` for unambiguous downstream routing
- [Phase 02-05] Fixed stale `processing_status` via `.dockerignore` exclusion (not runtime code change) — keeps Dockerfile/railway.toml/api invariants from Phase 01 untouched

**Issues Resolved:**
- I-1 (major): Empty FOLIO IRIs on LLM-path concept tags — root-caused to 0.7 threshold rejecting legitimate matches
- I-2 (minor): Bundle export 422 → 404 parity with sibling endpoints
- I-3 (blocker for viewer dev): Vite proxy target wrong port (8700 vs 9925)
- I-4 (test-coverage gap): No approved-task fixture for export validation
- I-5 (minor): Railway `test1` showed `processing_status=failed` (stale per-corpus `.jobs/` state bundled into image)
- Operational: Railway autodeploy had become disconnected (discovered during 02-05 verification) — user reconnected the service.source; future pushes auto-redeploy

- `output/default/` has no `corpus-meta.json` — catalog loader silently skips it
- Full UAT re-run against live URL for tests 14–15, 19, 24–28, 30, 33–36, 37–41, 47 deferred to `/gsd-verify-work 2`
