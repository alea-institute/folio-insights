<objective>
Fix UAT Issue I-5: Railway live `/api/v1/corpora` returns `test1` with `processing_status: "failed"`; local returns `completed`. Root cause (confirmed by inspection): the local `output/.jobs/` directory is bundled into the Docker image because `.dockerignore` does NOT exclude it. Whatever stale or failed-state job JSON happens to sit on disk when Railway's Dockerfile build runs gets baked into the image. The fix is a single-line `.dockerignore` entry that excludes `output/.jobs/` from the build context. Once stripped, `api/routes/corpus.py::_read_corpus_info` falls back to its "extraction.json exists → status=completed" branch (line ~74-76), which is the correct display state for the pre-processed bundled corpora.

Purpose: Remove the "failed" first impression from the live dev URL without changing runtime behavior. A visitor loading https://folio-insights-production.up.railway.app sees clean "completed" status for all bundled corpora.
Output: `.dockerignore` with a new `output/.jobs` exclusion line; a user-executed Railway redeploy + post-deploy curl verification that confirms the fix landed on the live URL.
</objective>

  processing_status = "not_processed"        # default
  job_path = output_dir / ".jobs" / f"{corpus_id}.json"
  if job_path.exists():
      # reads job file → status = completed / processing / failed / pending

      # Fallback: extraction.json exists but no job file
      processing_status = "completed"

  So: stripping .jobs/ from the image → on Railway, job_path.exists() == False →
  fallback triggers → bundled corpora with extraction.json (test1, demo) show "completed".
