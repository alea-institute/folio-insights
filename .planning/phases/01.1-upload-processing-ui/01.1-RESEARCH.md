## Summary

This phase adds a browser-based upload and processing workflow to the existing folio-insights viewer, replacing the CLI-only `folio-insights extract <dir>` flow. The implementation spans both the Python backend (FastAPI routes for upload, corpus CRUD, pipeline triggering, and SSE streaming) and the Svelte frontend (new Upload page with drag-drop zone, corpus sidebar, file list, Process button, SSE progress display, and auto-navigation to Review).

The project already has all core pipeline infrastructure in place: `PipelineOrchestrator` chains 7 stages with checkpointing, `CorpusRegistry` tracks files by SHA-256 hash, `IngestionStage` handles 14 formats, and the output formatter writes the JSON files consumed by the review viewer. The existing folio-enrich codebase provides a battle-tested reference implementation for SSE streaming (`sse.py`), job management (`job_store.py`), and frontend SSE consumption (`startSSE` in `index.html`). The key work is: (1) new FastAPI routes for file upload + corpus CRUD + processing trigger + SSE stream, (2) adapting the existing synchronous `PipelineOrchestrator.run()` to emit progress events consumable by an SSE poller, (3) a new SvelteKit Upload page with all the components specified in the UI-SPEC.

**Primary recommendation:** Model the backend closely after folio-enrich's proven pattern -- `asyncio.create_task()` for background pipeline execution, disk-based job persistence with polling SSE generator, `sse-starlette` for EventSourceResponse. Frontend uses native `EventSource` API with typed event listeners. Use `multipart/FormData` for uploads (already decided). Add `sse-starlette` and `python-multipart` as new Python dependencies.

### Pattern 1: Background Pipeline with Polling SSE (from folio-enrich)

**What:** Pipeline runs in an `asyncio.create_task()` background task. A ProcessingJob object persists to disk. The SSE endpoint polls the job file every 0.5s and yields events for status changes, activity log entries, and completion.

**When to use:** Always -- this is the proven pattern from folio-enrich.

**Example (backend):**
```python
# Source: folio-enrich/backend/app/api/routes/enrich.py
from sse_starlette.sse import EventSourceResponse

@router.post("/api/v1/corpus/{corpus_id}/process", status_code=202)
async def start_processing(corpus_id: str):
    job = ProcessingJob(corpus_id=corpus_id, status=ProcessingStatus.PENDING)
    await job_manager.save(job)
    asyncio.create_task(run_pipeline(job))
    return {"job_id": str(job.id), "status": job.status.value}

@router.get("/api/v1/corpus/{corpus_id}/stream")
async def stream_progress(corpus_id: str):
    return EventSourceResponse(processing_event_stream(corpus_id, job_manager))
```

**Example (SSE generator adapted from folio-enrich):**
```python
# Source: adapted from folio-enrich/backend/app/services/streaming/sse.py
async def processing_event_stream(corpus_id, job_manager, poll_interval=0.5):
    last_status = None
    last_activity_count = 0

    while True:
        job = await job_manager.load_by_corpus(corpus_id)
        if job is None:
            yield {"event": "error", "data": json.dumps({"error": "Job not found"})}
            return

        if job.status != last_status:
            last_status = job.status
            yield {
                "event": "status",
                "data": json.dumps({
                    "job_id": str(job.id),
                    "status": job.status.value,
                    "stage": job.current_stage,
                    "progress": job.progress_pct,
                }),
            }

        # Emit new activity log entries
        if len(job.activity_log) > last_activity_count:
            for entry in job.activity_log[last_activity_count:]:
                yield {"event": "activity", "data": json.dumps(entry)}
            last_activity_count = len(job.activity_log)

        if job.status in (ProcessingStatus.COMPLETED, ProcessingStatus.FAILED):
            yield {
                "event": "complete",
                "data": json.dumps({
                    "status": job.status.value,
                    "total_units": job.total_units,
                    "error": job.error,
                }),
            }
            return

        await asyncio.sleep(poll_interval)
```

### Pattern 2: Adapted PipelineOrchestrator with Progress Callbacks

**What:** The existing `PipelineOrchestrator.run()` method is synchronous-ish (awaits stages sequentially). To emit SSE events, the orchestrator needs to update a shared ProcessingJob object after each stage completes. This avoids modifying the orchestrator's core logic -- instead, wrap each stage execution with a callback that updates the job store.

**When to use:** When connecting the existing pipeline to the SSE system.

**Example:**
```python
async def run_pipeline(job: ProcessingJob):
    """Run pipeline in background task, updating job state for SSE polling."""
    settings = get_settings()
    orchestrator = PipelineOrchestrator(settings)

    stages = ["ingestion", "structure", "boundaries", "distillation",
              "classification", "folio_tagging", "deduplication"]

    try:
        for i, stage in enumerate(orchestrator._stages):
            job.current_stage = stage.name
            job.status = ProcessingStatus.PROCESSING
            job.progress_pct = int((i / len(stages)) * 100)
            job.activity_log.append({
                "timestamp": datetime.now(timezone.utc).isoformat(),
                "stage": stage.name,
                "message": f"Starting {stage.name}...",
            })
            await job_manager.save(job)

            insights_job = await stage.execute(insights_job)

            job.activity_log.append({
                "timestamp": datetime.now(timezone.utc).isoformat(),
                "stage": stage.name,
                "message": f"Completed {stage.name} ({len(insights_job.units)} units)",
            })
            await job_manager.save(job)

        # Post-pipeline output
        job.status = ProcessingStatus.COMPLETED
        job.progress_pct = 100
        job.total_units = len(insights_job.units)
        await job_manager.save(job)

    except Exception as e:
        job.status = ProcessingStatus.FAILED
        job.error = str(e)
        await job_manager.save(job)
```

**What:** Use native `EventSource` API with typed event listeners. Handle reconnection states. Store SSE connection reference for cleanup.

**Example:**

// Source: adapted from folio-enrich/frontend/index.html startSSE()
function connectSSE(corpusId: string): EventSource {
  const es = new EventSource(`${API_BASE}/api/v1/corpus/${corpusId}/stream`);

  es.addEventListener('status', (e) => {
    const data = JSON.parse(e.data);
    updateProgress(data.status, data.stage, data.progress);
  });

  es.addEventListener('activity', (e) => {
    const data = JSON.parse(e.data);
    appendActivityLog(data);
  });

  es.addEventListener('complete', (e) => {
    const data = JSON.parse(e.data);
    handleCompletion(data);
    es.close();
  });

  es.onerror = () => {
    // EventSource auto-reconnects; update UI state
    updateConnectionState('reconnecting');
  };

  return es;
}

### Anti-Patterns to Avoid

- **Modifying PipelineOrchestrator internals:** Don't refactor the orchestrator's stage execution loop. Instead, wrap it in a runner function that updates the ProcessingJob between stages.
- **Storing uploaded files in memory:** Write files to disk immediately in the upload endpoint. The corpus source directory is the single source of truth.
- **Polling from the frontend:** Don't use `setInterval` + `fetch` for progress. Use SSE (EventSource) which handles reconnection automatically.
- **Building a custom SSE implementation:** Use `sse-starlette`'s `EventSourceResponse` -- it handles keep-alive, proper headers, and connection management.
- **Breaking the existing review page:** The Upload page is additive. The review page at `/` (or `/review`) must continue working exactly as before.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSE response formatting | Custom streaming response with manual headers | `sse-starlette.EventSourceResponse` | Handles keep-alive, proper `text/event-stream` content type, connection lifecycle |
| File upload parsing | Manual multipart parsing | FastAPI `UploadFile` + `python-multipart` | Handles streaming, temp files, content type detection |
| ZIP extraction | Custom archive handling | `zipfile.ZipFile` (stdlib) | Handles all ZIP variants, path traversal protection with `extractall()` |
| SSE client reconnection | Custom retry logic with fetch | Native `EventSource` API | Auto-reconnects with exponential backoff built into browsers |
| SPA routing | Hash-based routing or manual history API | SvelteKit file-based routing | Already configured with adapter-static fallback |
| SHA-256 file hashing | Custom implementation | Existing `CorpusRegistry._compute_hash()` | Already implemented and tested in the project |
| Pipeline stage execution | New pipeline runner | Existing `PipelineOrchestrator.run()` | Already chains all 7 stages with checkpointing |

**Key insight:** The entire pipeline infrastructure already exists. The work is plumbing: connecting uploads to the source directory, triggering the orchestrator as a background task, and streaming its progress back to the browser.

## Common Pitfalls

### Pitfall 1: ZIP Path Traversal (Zip Slip)
**What goes wrong:** Malicious ZIP files contain entries with `../` paths that write outside the extraction directory.
**Why it happens:** Naive `zipfile.extractall()` without validation.
**How to avoid:** Validate each entry's resolved path starts with the target directory. Python's `zipfile` module since 3.11.4 has some protections, but explicit checking is safer:
```python
for info in zf.infolist():
    target = extract_dir / info.filename
    if not str(target.resolve()).startswith(str(extract_dir.resolve())):
        raise ValueError(f"Zip entry escapes target directory: {info.filename}")
```
**Warning signs:** ZIP entries with `..` in path components.

### Pitfall 2: SSE Connection Leaks
**What goes wrong:** SSE generator keeps running after the client disconnects, consuming server resources.
**Why it happens:** The `while True` loop in the SSE generator doesn't detect client disconnection.
**How to avoid:** `sse-starlette` handles this automatically -- when the client closes the connection, the generator receives a `GeneratorExit` or `asyncio.CancelledError`. The folio-enrich pattern (terminal states cause `return`) also ensures the generator exits.
**Warning signs:** Growing memory usage, orphaned asyncio tasks.

### Pitfall 3: Race Condition Between Pipeline Task and SSE Poller
**What goes wrong:** The SSE poller reads the job file while the pipeline task is writing it, getting partial/corrupt JSON.
**Why it happens:** Non-atomic file writes.
**How to avoid:** Use atomic write (write to temp file, then rename) -- this is exactly what folio-enrich's `JobStore.save()` does. Replicate this pattern.
**Warning signs:** Occasional JSON parse errors in the SSE generator.

### Pitfall 4: Blocking the Event Loop During Pipeline Execution
**What goes wrong:** CPU-intensive pipeline stages (e.g., boundary detection with sentence-transformers) block the event loop, making SSE polling and file uploads unresponsive.
**Why it happens:** `asyncio.create_task()` runs in the same event loop; CPU-bound work blocks it.
**How to avoid:** The existing pipeline stages already use `await` for I/O-bound work (LLM calls via `instructor`). For CPU-intensive stages like sentence-transformers embedding, consider wrapping in `asyncio.to_thread()` if responsiveness is an issue. However, since folio-enrich uses the same pattern successfully, this may be acceptable as-is.
**Warning signs:** SSE events arriving in bursts rather than smoothly during heavy computation stages.

### Pitfall 5: CORS Issues with SSE in Dev Mode
**What goes wrong:** EventSource connections fail with CORS errors during development (Vite dev server on :5173 connecting to FastAPI on :8700).
**Why it happens:** SSE uses a plain GET request, but CORS preflight may not be configured for `text/event-stream`.
**How to avoid:** The Vite proxy at `/api` already forwards to FastAPI. Use relative URLs (`/api/v1/corpus/{id}/stream`) in dev mode via the proxy, not direct cross-origin connections. The existing `API_BASE` pattern in `client.ts` handles this.
**Warning signs:** SSE works in production but fails in dev mode.

### Pitfall 6: Large File Upload Timeouts
**What goes wrong:** Uploading large files (e.g., 50MB ZIP archives) times out.
**Why it happens:** Default server/proxy timeouts are too short.
**How to avoid:** FastAPI/Uvicorn default request timeout is generous (no default timeout for request body). The Vite proxy may need timeout configuration for large uploads. Set reasonable file size limits (e.g., 100MB per file, 500MB per upload batch).
**Warning signs:** Uploads of large files consistently fail.

### Pitfall 7: `webkitdirectory` Browser Compatibility
**What goes wrong:** Folder upload via `webkitdirectory` doesn't work in all browsers.
**Why it happens:** `webkitdirectory` is a non-standard attribute. Firefox and Chrome support it, but it's not universally supported.
**How to avoid:** Treat folder upload as progressive enhancement. Always have the file picker and drag-drop as fallbacks. Test in target browsers.
**Warning signs:** File input shows no files when a folder is selected in some browsers.

### Pitfall 8: SvelteKit Routing with FastAPI StaticFiles
**What goes wrong:** Navigating directly to `/upload` returns a 404 because FastAPI's `StaticFiles` tries to serve a literal file at that path.
**Why it happens:** SPA routing requires all non-file routes to return `index.html`.
**How to avoid:** The project already uses `adapter-static` with `fallback: 'index.html'` and mounts via `StaticFiles(directory=..., html=True)`. The `html=True` parameter tells Starlette to serve `index.html` for directory requests, and the fallback configuration generates a fallback page. However, verify that direct navigation to `/upload` works with the current mount. May need to add a catch-all route in FastAPI that serves `index.html` for non-API, non-file paths.
**Warning signs:** Direct URL navigation to `/upload` returns 404; only in-app navigation works.

### File Upload Endpoint (FastAPI)
```python
# Source: FastAPI docs + project conventions
from fastapi import APIRouter, UploadFile, File, HTTPException
from pathlib import Path
import shutil

router = APIRouter(prefix="/api/v1", tags=["upload"])

@router.post("/corpus/{corpus_id}/upload")
async def upload_files(
    corpus_id: str,
    files: list[UploadFile] = File(...),
):
    corpus_dir = _get_corpus_source_dir(corpus_id)
    corpus_dir.mkdir(parents=True, exist_ok=True)

    uploaded = []
    for file in files:
        dest = corpus_dir / file.filename
        with open(dest, "wb") as f:
            shutil.copyfileobj(file.file, f)
        uploaded.append({"filename": file.filename, "size": dest.stat().st_size})

    return {"uploaded": uploaded, "count": len(uploaded)}
```

### SSE Event Consumption (Svelte store)
```typescript
// Source: adapted from folio-enrich frontend + Svelte store pattern
import { writable, get } from 'svelte/store';

export const processingStatus = writable<'idle' | 'processing' | 'complete' | 'error'>('idle');
export const currentStage = writable<string>('');
export const progressPct = writable<number>(0);
export const activityLog = writable<ActivityEntry[]>([]);
export const sseState = writable<'connected' | 'reconnecting' | 'closed'>('closed');

let eventSource: EventSource | null = null;

export function startProcessingStream(corpusId: string) {
    closeStream();
    processingStatus.set('processing');
    sseState.set('connected');

    const base = import.meta.env.DEV ? 'http://localhost:8700' : '';
    eventSource = new EventSource(`${base}/api/v1/corpus/${corpusId}/stream`);

    eventSource.addEventListener('status', (e) => {
        const data = JSON.parse(e.data);
        currentStage.set(data.stage);
        progressPct.set(data.progress);
    });

    eventSource.addEventListener('activity', (e) => {
        const data = JSON.parse(e.data);
        activityLog.update(log => [...log, data]);
    });

    eventSource.addEventListener('complete', (e) => {
        const data = JSON.parse(e.data);
        processingStatus.set(data.status === 'completed' ? 'complete' : 'error');
        sseState.set('closed');
        eventSource?.close();
    });

    eventSource.onerror = () => {
        sseState.set('reconnecting');
    };
}

export function closeStream() {
    eventSource?.close();
    eventSource = null;
    sseState.set('closed');
}
```

## Open Questions

1. **Corpus ID Format**
   - What we know: Corpora need unique identifiers for URL paths and file system directories
   - What's unclear: Whether to use user-provided names directly (URL-encoded) or generate slugs/UUIDs
   - Recommendation: Use slugified version of user-provided name (lowercase, hyphens) as the corpus ID. This makes URLs readable and file system paths clean. Validate uniqueness on creation.

2. **Pipeline Orchestrator Modification Scope**
   - What we know: The existing `PipelineOrchestrator.run()` takes a `source_dir` and `corpus_name`, returns an `InsightsJob`. It doesn't have progress callbacks.
   - What's unclear: Whether to modify `PipelineOrchestrator` to accept a callback, or to build a wrapper that calls individual stages and updates the job between them.
   - Recommendation: Build a thin wrapper function (`run_pipeline_with_progress()`) that accesses `orchestrator._stages` directly and runs them sequentially with job updates between each stage. This avoids modifying the tested orchestrator code.

3. **Corpus Metadata Storage**
   - What we know: The project uses aiosqlite for review persistence and JSON files for extraction data. Corpus manifests are JSON files.
   - What's unclear: Whether to add a SQLite table for corpus metadata or keep using JSON manifest files.
   - Recommendation: Use JSON manifest files (already established by `CorpusRegistry`). Add a `corpora.json` index file in the output directory listing all corpora. This matches the existing pattern and avoids database schema migration complexity.

4. **Review Page Corpus Selector Integration**
   - What we know: The review page has a static `<select>` with only "default" option. It needs to be populated from the corpus list API.
   - What's unclear: How to share corpus state between Upload and Review pages.
   - Recommendation: Create a shared `corpus` store in `$lib/stores/corpus.ts` that both pages use. The `+layout.svelte` corpus selector reads from this store and populates from the API on mount.
